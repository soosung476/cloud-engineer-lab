# Linux 운영 명령어 정리

## 1. 문서 목적

이 문서는 EC2 Ubuntu 서버를 운영하면서 실제로 사용한  
기본 Linux 명령어를 정리한 문서다.

단순히 명령어 뜻만 적는 것이 아니라,  
**언제 사용하는지**, **무엇을 확인하는지**,  
**운영 관점에서 왜 중요한지**를 함께 기록한다.

이 문서의 목적은  
시험용 암기가 아니라  
**실제 서버 운영 중 장애 확인과 상태 점검에 바로 사용할 수 있는 기준**을 만드는 것이다.

---

## 2. 서비스 관리 명령어

---

### `systemctl status nginx`

#### 목적
nginx 서비스의 현재 상태를 확인한다.

#### 언제 사용하는가
- 웹 접속이 되지 않을 때
- 서비스가 정상 실행 중인지 확인할 때
- stop / start 이후 상태를 확인할 때

#### 확인 포인트
- `active (running)` 인지
- `inactive (dead)` 인지
- 에러 메시지가 있는지
- 서비스가 systemd에 의해 정상 관리되는지

#### 운영 관점에서 중요한 이유
웹 서비스 장애가 발생했을 때  
가장 먼저 확인해야 하는 항목 중 하나가 서비스 상태다.

---

### `systemctl stop nginx`

#### 목적
nginx 서비스를 중지한다.

#### 언제 사용하는가
- 서비스 중지 테스트
- 장애 상황 재현
- 설정 변경 전 임시 중지

#### 확인 포인트
- stop 이후 상태가 실제로 inactive로 바뀌는지
- 브라우저 접속이 실패하는지

#### 운영 관점에서 중요한 이유
서비스가 멈추면 사용자 접속도 즉시 영향을 받는다.  
서비스 중지가 실제 서비스 장애와 어떻게 연결되는지 이해할 수 있다.

---

### `systemctl start nginx`

#### 목적
중지된 nginx 서비스를 다시 시작한다.

#### 언제 사용하는가
- stop 이후 복구할 때
- 서비스가 죽어 있을 때 다시 올릴 때

#### 확인 포인트
- active 상태로 변경되는지
- 브라우저 접속이 복구되는지

#### 운영 관점에서 중요한 이유
운영자는 단순히 장애를 발견하는 것뿐 아니라  
서비스를 복구할 수 있어야 한다.

---

## 3. 로그 확인 명령어

---

### `journalctl -u nginx --no-pager | tail -n 30`

#### 목적
nginx 서비스의 최근 로그를 확인한다.

#### 언제 사용하는가
- nginx 실행 실패 원인을 확인할 때
- start / restart 이후 문제가 있는지 확인할 때
- 서비스 관련 메시지를 빠르게 보고 싶을 때

#### 확인 포인트
- 포트 바인딩 실패
- 설정 오류
- 권한 관련 문제
- systemd 서비스 메시지

#### 운영 관점에서 중요한 이유
로그는 장애 원인을 가장 직접적으로 보여주는 자료다.  
운영자는 “안 된다”는 현상만 보는 것이 아니라  
**왜 안 되는지**를 로그에서 찾을 수 있어야 한다.

---

### `tail -n 30 /var/log/nginx/access.log`

#### 목적
nginx access log의 최근 요청 기록을 확인한다.

#### 언제 사용하는가
- 실제 HTTP 요청이 들어오는지 확인할 때
- 브라우저 접속 요청이 서버에 도달했는지 확인할 때

#### 확인 포인트
- 최근 요청 시간
- 클라이언트 요청 여부
- HTTP 응답 코드

#### 운영 관점에서 중요한 이유
서비스가 살아 있어도 요청이 실제로 들어오는지,  
그리고 정상 응답하는지는 access log를 통해 확인할 수 있다.

---

### `tail -n 30 /var/log/nginx/error.log`

#### 목적
nginx error log의 최근 에러 메시지를 확인한다.

#### 언제 사용하는가
- 웹 페이지가 비정상 동작할 때
- nginx 내부 오류를 확인할 때
- 설정 또는 파일 문제를 추적할 때

#### 확인 포인트
- permission 에러
- file not found
- 설정 관련 오류
- 기타 런타임 오류

#### 운영 관점에서 중요한 이유
에러 로그는 서비스가 실패하는 직접 원인을 보여준다.  
access log와 error log를 함께 보는 습관이 중요하다.

---

## 4. 프로세스 / 포트 확인 명령어

---

### `ps -ef | grep nginx`

#### 목적
현재 nginx 관련 프로세스가 실행 중인지 확인한다.

#### 언제 사용하는가
- nginx가 실제로 떠 있는지 확인할 때
- 마스터 / 워커 프로세스 상태를 볼 때

#### 확인 포인트
- nginx 프로세스 존재 여부
- master process / worker process 확인

#### 운영 관점에서 중요한 이유
서비스 상태가 정상처럼 보여도 실제 프로세스 상태를 함께 보는 것이 좋다.  
프로세스는 서비스 운영의 가장 기본 단위다.

---

### `ss -tulpn | grep :80`

#### 목적
80번 포트를 어떤 프로세스가 리슨 중인지 확인한다.

#### 언제 사용하는가
- 브라우저 접속이 안 될 때
- nginx가 실제로 80 포트를 열고 있는지 볼 때
- 포트 충돌 가능성을 확인할 때

#### 확인 포인트
- `LISTEN` 상태 여부
- 80 포트를 사용하는 프로세스가 nginx인지 여부

#### 운영 관점에서 중요한 이유
서비스가 실행 중이더라도 실제로 포트를 열지 못하면 접속은 실패한다.  
포트 확인은 웹 장애 점검의 핵심이다.

---

## 5. 시스템 자원 확인 명령어

---

### `df -h`

#### 목적
디스크 사용량을 확인한다.

#### 언제 사용하는가
- 디스크 공간 부족 여부를 확인할 때
- 로그 파일 증가로 용량 문제가 없는지 볼 때

#### 확인 포인트
- 전체 용량
- 사용량
- 남은 용량
- 사용률(%)

#### 운영 관점에서 중요한 이유
디스크가 꽉 차면 로그 기록, 패키지 설치, 서비스 동작에 문제가 생길 수 있다.  
운영에서는 자원 확인도 매우 중요하다.

---

### `free -h`

#### 목적
메모리 사용량을 확인한다.

#### 언제 사용하는가
- 시스템이 느리다고 느껴질 때
- 메모리 부족 여부를 점검할 때

#### 확인 포인트
- total / used / free memory
- available memory
- swap 사용 여부

#### 운영 관점에서 중요한 이유
메모리는 서비스 성능과 안정성에 직접 영향을 준다.  
운영자는 CPU뿐 아니라 메모리 상태도 함께 봐야 한다.

---

## 6. 실제 점검 흐름 예시

웹 페이지 접속이 안 될 때는 아래 순서로 확인한다.

### 1단계: 서비스 상태 확인
- `systemctl status nginx`

### 2단계: 프로세스 확인
- `ps -ef | grep nginx`

### 3단계: 포트 확인
- `ss -tulpn | grep :80`

### 4단계: 서비스 로그 확인
- `journalctl -u nginx --no-pager | tail -n 30`

### 5단계: 웹 서버 로그 확인
- `tail -n 30 /var/log/nginx/access.log`
- `tail -n 30 /var/log/nginx/error.log`

### 6단계: 시스템 자원 확인
- `df -h`
- `free -h`

이 흐름은 단순 암기가 아니라  
실제 운영 장애 상황에서 빠르게 원인을 좁혀가기 위한 기본 절차다.

---

## 7. 이번 실습에서 배운 점

- Linux 명령어는 따로 외우는 것이 아니라 상황과 함께 익혀야 오래 남는다.
- `systemctl`, `journalctl`, `ps`, `ss`는 서비스 장애 점검의 핵심 도구다.
- access log와 error log를 함께 봐야 웹 서비스 상태를 더 정확히 이해할 수 있다.
- 디스크와 메모리 확인도 운영의 중요한 기본기다.
