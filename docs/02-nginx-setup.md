# nginx 설치 및 서비스 제어

## 1. 실습 목적

EC2 Ubuntu 인스턴스에 nginx를 설치하고,  
웹 서버가 정상적으로 실행되는지 확인한다.

또한 systemd 기반 서비스 제어와  
웹 서버 장애 발생 시 기본적으로 어떤 항목을 확인해야 하는지 익힌다.

이 실습의 핵심은  
**단순 설치가 아니라 “서비스를 운영 가능한 상태로 보고 제어하는 경험”**을 쌓는 것이다.

---

## 2. 실습 환경

- Cloud: AWS EC2
- OS: Ubuntu
- Web Server: nginx
- Access:
  - SSH 접속
  - 브라우저 HTTP 접속

---

## 3. 실습 목표

이번 실습에서 확인한 목표는 다음과 같다.

- nginx 설치
- nginx 실행 상태 확인
- systemctl로 서비스 상태 조회
- stop / start 실습
- 브라우저 접속 확인
- 기본 페이지를 커스텀 HTML로 교체

---

## 4. 작업 과정

### 4-1. 패키지 업데이트
먼저 패키지 목록을 최신 상태로 반영했다.

이 단계는 설치 실패 가능성을 줄이고,  
최신 패키지 정보를 기준으로 작업하기 위해 필요하다.

### 4-2. nginx 설치
Ubuntu 패키지 관리자를 통해 nginx를 설치했다.

설치 후 확인한 것:
- nginx 패키지 설치 완료 여부
- 서비스 등록 여부

### 4-3. 서비스 상태 확인
`systemctl status nginx` 명령어로 nginx 상태를 확인했다.

여기서 확인한 핵심:
- 서비스가 active 상태인지
- systemd가 서비스 프로세스를 정상 관리 중인지
- 에러 메시지가 있는지

### 4-4. 브라우저 접속 테스트
EC2 퍼블릭 IP로 접속하여 기본 nginx 페이지가 보이는지 확인했다.

이 단계는 단순히 nginx가 실행 중인지뿐 아니라,
아래 조건이 모두 충족되었는지 함께 검증하는 의미가 있다.

- EC2 실행 중
- nginx 정상 실행 중
- 80 포트 리스닝 중
- Security Group에서 HTTP 허용
- 브라우저에서 외부 접속 가능

### 4-5. stop / start 실습
운영 중인 서비스를 일부러 중지하고 다시 시작해보며,
서비스 제어 흐름을 확인했다.

실습 내용:
- nginx stop
- 상태 확인
- 웹 접속 실패 확인
- nginx start
- 상태 복구 확인
- 웹 접속 복구 확인

이 과정을 통해  
**서비스 중단이 사용자 접속 실패로 바로 이어진다**는 점을 눈으로 확인했다.

### 4-6. 기본 HTML 페이지 교체
기본 nginx welcome 페이지를 내 HTML 파일로 교체했다.

이 과정에서 확인한 것:
- 웹 루트 경로에 파일 반영
- 브라우저 새로고침 후 반영 여부
- 정적 콘텐츠 배포의 가장 기본적인 구조

---

## 5. 확인 결과

이번 실습을 통해 아래 항목을 확인했다.

- nginx 설치 성공
- nginx 서비스 정상 실행 확인
- systemctl 기반 서비스 상태 조회 가능
- stop / start에 따라 웹 접속 가능 여부가 달라짐을 확인
- 기본 페이지를 내 HTML로 교체 완료

즉,  
**서버 한 대에 웹 서버를 올리고, 서비스 실행 여부에 따라 실제 외부 접속이 달라지는 흐름을 직접 경험했다.**

---

## 6. 운영 관점에서 중요한 이유

클라우드 운영 / 시스템 엔지니어 입장에서 nginx 설치 실습은 단순한 웹 서버 세팅이 아니다.

이 실습에서 중요한 운영 포인트는 다음과 같다.

- 서비스는 설치만으로 끝나는 것이 아니라 상태 점검이 필요하다.
- 사용자는 브라우저로 “접속 가능/불가”만 보지만, 운영자는 그 뒤의 원인을 추적해야 한다.
- 서비스 중지, 포트 차단, 보안 그룹, 파일 문제는 모두 동일하게 “웹 접속 안 됨”으로 보일 수 있다.
- 따라서 장애 시에는 서비스 상태, 포트, 로그, 네트워크를 순서대로 확인해야 한다.

---

## 7. 장애 상황에서 확인할 항목

웹 페이지가 열리지 않을 때는 아래 순서로 확인한다.

1. EC2 인스턴스가 Running 상태인지 확인
2. `systemctl status nginx`로 서비스 상태 확인
3. `ss -tulpn | grep :80` 으로 80 포트 리스닝 확인
4. Security Group에서 80 포트 허용 여부 확인
5. `journalctl -u nginx --no-pager | tail -n 30` 으로 서비스 로그 확인
6. `/var/log/nginx/access.log`, `/var/log/nginx/error.log` 확인
7. 웹 루트 경로의 HTML 파일 상태 및 권한 확인

---

## 8. 발생할 수 있는 문제

### nginx 서비스가 inactive 상태
원인 예시:
- 수동으로 stop 한 상태
- 부팅 후 자동 시작이 설정되지 않음
- 설정 파일 오류로 실행 실패

### 브라우저 접속 불가
원인 예시:
- Security Group에서 80 미허용
- nginx 미실행
- 80 포트 리스닝 실패
- 퍼블릭 IP 확인 오류

### HTML 변경이 반영되지 않음
원인 예시:
- 파일 경로 잘못 지정
- 파일 덮어쓰기 실패
- 브라우저 캐시
- 권한 문제

---

## 9. 트러블슈팅 예시

### 상황
브라우저에서 페이지가 열리지 않음

### 확인 순서
1. EC2 상태 확인
2. nginx 서비스 상태 확인
3. 80 포트 리스닝 확인
4. Security Group 확인
5. nginx 로그 확인

### 핵심 교훈
웹 접속 실패는 단순히 “nginx 문제”가 아닐 수 있다.  
서버 상태, 포트, 보안 그룹, 로그를 함께 봐야 정확한 원인을 찾을 수 있다.

---

## 10. 배운 점

- 서비스 운영에서는 설치보다 상태 확인과 복구가 더 중요하다.
- `systemctl`은 서비스 제어의 핵심 명령어다.
- 웹 접속 장애는 서비스/포트/보안/로그를 함께 확인해야 한다.
- HTML 교체를 통해 정적 웹 콘텐츠 배포의 가장 기본적인 흐름을 익혔다.
