# Security Group 점검 및 실습

## 1. 실습 목적

AWS Security Group이 EC2 인스턴스의 외부 접속 가능 여부에 어떤 영향을 주는지 확인한다.

이번 실습의 목적은 단순히 포트를 여닫는 것이 아니라,  
**클라우드 환경에서 네트워크 접근 제어가 실제 서비스 운영에 어떻게 작동하는지 이해하는 것**이다.

특히 아래 항목을 중점적으로 확인했다.

- SSH(22) 접속 제어
- HTTP(80) 접속 제어
- 인바운드 규칙 변경에 따른 접속 성공/실패 변화
- 서비스가 정상 동작해도 Security Group이 막으면 접속할 수 없다는 점

---

## 2. 실습 환경

- Cloud: AWS EC2
- OS: Ubuntu
- Web Server: nginx
- Access Method:
  - SSH
  - Browser HTTP access
- Local Environment: MacBook M1

---

## 3. 실습 목표

이번 실습에서 확인한 목표는 다음과 같다.

- Security Group의 역할 이해
- 인바운드 규칙과 외부 접속의 관계 확인
- 22번 포트의 의미 이해
- 80번 포트의 의미 이해
- HTTP 포트 제거 시 웹 접속 실패 확인
- HTTP 포트 복구 시 웹 접속 정상화 확인
- SSH 포트를 가능한 범위에서 제한하는 이유 이해

---

## 4. Security Group 개념 정리

Security Group은 AWS에서 EC2 인스턴스에 적용하는  
**가상 방화벽(Virtual Firewall)** 역할을 한다.

즉, 인스턴스 자체가 살아 있고 nginx 서비스도 정상 실행 중이어도  
Security Group에서 필요한 포트를 허용하지 않으면 외부에서는 접속할 수 없다.

이번 실습에서 사용한 핵심 포트는 다음과 같다.

### SSH (22/tcp)
원격 터미널 접속을 위한 포트다.  
운영자가 서버에 로그인해서 작업할 때 사용한다.

### HTTP (80/tcp)
브라우저를 통해 웹 페이지에 접속할 때 사용하는 포트다.  
nginx 웹 서버가 외부 사용자 요청을 받을 때 사용한다.

---

## 5. 작업 과정

### 5-1. 현재 Security Group 규칙 확인
먼저 EC2 인스턴스에 연결된 Security Group의 인바운드 규칙을 확인했다.

초기 상태에서 확인한 주요 항목:
- SSH 22 허용
- HTTP 80 허용

이 단계에서 Security Group이  
EC2 외부 접근 경로의 첫 번째 제어 지점이라는 점을 다시 확인했다.

---

### 5-2. HTTP(80) 허용 상태에서 웹 접속 확인
브라우저에서 EC2의 퍼블릭 IP로 접속하여  
nginx 웹 페이지가 정상적으로 열리는지 확인했다.

이 단계에서 확인한 의미는 단순히 페이지가 열린다는 것이 아니라,  
다음 조건이 모두 만족된다는 뜻이다.

- EC2 인스턴스가 Running 상태
- nginx 서비스가 정상 실행 중
- 80번 포트가 리스닝 중
- Security Group에서 HTTP(80) 허용
- 브라우저에서 외부 접근 가능

---

### 5-3. HTTP(80) 인바운드 규칙 제거
이후 Security Group에서 80번 포트 허용 규칙을 제거했다.

이 작업의 목적은  
**웹 서버가 정상이어도 네트워크 접근 제어 정책에 의해 외부 접속이 차단될 수 있는지 직접 확인하는 것**이다.

---

### 5-4. HTTP 제거 후 웹 접속 실패 확인
80번 포트를 제거한 뒤 브라우저로 다시 접속을 시도했다.

결과적으로 웹 페이지에 정상 접속할 수 없었고,  
이를 통해 다음을 확인할 수 있었다.

- nginx 서비스 상태와 관계없이
- Security Group에서 HTTP를 차단하면
- 외부 브라우저 접속은 실패한다

즉,  
**서비스 상태와 네트워크 접근 정책은 별개의 계층**이며,  
운영자는 둘 다 확인해야 한다는 점을 실습으로 이해했다.

---

### 5-5. HTTP(80) 규칙 복구 후 재확인
이후 80번 포트 허용 규칙을 다시 추가하고  
브라우저에서 재접속을 시도했다.

웹 페이지가 다시 정상적으로 열리는 것을 확인했고,  
이를 통해 Security Group 규칙 변경이 외부 접속에 직접적인 영향을 준다는 점을 명확히 확인했다.

---

### 5-6. SSH(22) 규칙 점검
SSH 접속용 22번 포트 규칙도 함께 점검했다.

이번 실습을 통해, SSH는 웹 공개용 포트와 달리  
가능하면 `0.0.0.0/0` 전체 공개보다 **내 IP만 허용하는 방식이 더 안전하다**는 점을 이해했다.

이유는 다음과 같다.

- SSH는 관리자 접속용 포트이기 때문
- 불필요한 외부 스캔 및 접근 시도를 줄일 수 있기 때문
- 최소 권한 원칙에 맞기 때문

---

## 6. 확인 결과

이번 실습을 통해 아래 내용을 확인했다.

- Security Group은 EC2 외부 접속을 제어하는 핵심 보안 장치다
- nginx 서비스가 정상 실행 중이어도 80 포트가 막히면 웹 접속은 실패한다
- HTTP 허용 규칙을 제거하면 브라우저 접속이 불가능해진다
- HTTP 허용 규칙을 복구하면 웹 접속이 다시 정상화된다
- SSH 포트는 가능한 한 내 IP 기준으로 제한하는 것이 더 안전하다

즉,  
**클라우드 운영에서는 서비스 자체의 상태뿐 아니라 네트워크 접근 제어 정책도 반드시 함께 확인해야 한다**는 점을 직접 확인했다.

---

## 7. 운영 관점에서 중요한 이유

클라우드 운영 / 시스템 엔지니어 관점에서  
“웹이 안 열린다”, “SSH가 안 된다”는 현상은 매우 흔하다.

이때 원인은 단순히 서버 문제만이 아닐 수 있다.

예를 들어 아래처럼 원인이 나뉠 수 있다.

- EC2 인스턴스 자체가 중지됨
- nginx 서비스가 중지됨
- 80 포트 리스닝 실패
- Security Group 인바운드 규칙 차단
- 잘못된 퍼블릭 IP 사용
- SSH 포트 미허용

이번 실습은 이 중에서  
**Security Group이 독립적인 장애 원인이 될 수 있다**는 점을 보여준다.

즉,  
운영자는 서비스 상태만 보는 것이 아니라  
네트워크 접근 정책도 함께 점검할 수 있어야 한다.

## 8. 발생할 수 있는 문제

### 웹 페이지가 열리지 않는 경우
가능한 원인:
- Security Group에서 80 포트 미허용
- nginx 서비스 중지
- EC2 인스턴스 중지
- 80 포트 리스닝 실패
- 퍼블릭 IP 확인 오류

### SSH 접속이 되지 않는 경우
가능한 원인:
- Security Group에서 22 포트 미허용
- SSH 허용 범위가 내 현재 IP와 다름
- 퍼블릭 IP 확인 오류
- 키 파일 권한 문제
- 사용자명 오류

### 보안상 좋지 않은 설정
예시:
- SSH 22 포트를 `0.0.0.0/0` 로 전체 공개
- 사용하지 않는 포트를 계속 열어둠

---

## 9. 트러블슈팅 체크리스트

### 웹 접속이 되지 않을 때
1. EC2 인스턴스가 Running 상태인지 확인
2. Security Group에서 80 포트가 허용되어 있는지 확인
3. `systemctl status nginx`로 서비스 상태 확인
4. `ss -tulpn | grep :80` 으로 80 포트 리스닝 확인
5. 퍼블릭 IP가 현재 인스턴스와 일치하는지 확인
6. `journalctl -u nginx --no-pager | tail -n 30` 으로 nginx 로그 확인
7. `/var/log/nginx/access.log`, `/var/log/nginx/error.log` 확인

### SSH 접속이 되지 않을 때
1. EC2 인스턴스가 Running 상태인지 확인
2. Security Group에서 22 포트가 허용되어 있는지 확인
3. 허용 소스가 현재 내 IP와 일치하는지 확인
4. 현재 퍼블릭 IP가 맞는지 확인
5. `.pem` 키 파일 권한 확인
6. Ubuntu 인스턴스라면 사용자명이 `ubuntu`인지 확인

---

## 10. 배운 점

- Security Group은 AWS에서 EC2를 보호하는 기본 네트워크 보안 장치다.
- 서비스가 살아 있어도 포트가 막히면 외부에서는 접속할 수 없다.
- 접속 장애는 서버 문제일 수도 있고, 보안 정책 문제일 수도 있다.
- SSH는 가능한 한 내 IP만 허용하는 방식이 더 안전하다.
- 운영에서는 “서버 상태 + 포트 + 보안그룹 + 로그”를 함께 봐야 정확한 원인을 찾을 수 있다.
